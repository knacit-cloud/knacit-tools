# 設計判断の記録

「なぜそう決めたか」を残す場所。新しい判断は上に追記する。

## 2026-07-17 — 第1弾は「方法論＋Skill」に決定（MCP案からの転換）

- **最初の公開物は、コードのツールではなく「クロス分析の方法論」と「それを実行するSkill」にする。**
  理由: GitHubでは「動くコード」より「知識・方法論」の方が2〜4桁大きく届く実測（prompts.chat 165k★、claude-cookbooks 49k★ vs 日本系MCP最高20★）。かつ Knacit が証明すべきは「コードが書ける」ではなく「AIを深く分かっている」であり、方法論公開はそれと一直線。
- **検討した「日本の営業日・祝日リモートMCP」は保留（没ではない）。**
  理由: 機能需要は実在（祝日ライブラリ月66万DL・日本専用営業日パッケージは空白・中国版の先行成功例あり）だが、Knacitの看板との繋がりが薄い。方法論の後の2弾目以降の候補として残す。
- **`DIAGNOSIS-TEMPLATE.md`（診断レポートの実テンプレ）は公開しない。**
  理由: 営業の設計（無料版の厚み配分・刺さる一言・抵抗を協力に変える話法）が含まれるため。公開するのは方法論・思想・検証済み根拠まで。
- **methods/ は英語主・日本語全文併記。**
  理由: 方法論は世界に読まれてこそ価値が出るカテゴリのため英語を主とし、国内向けに日本語全文を同居させる。

## 2026-07-16 — 複数ツールの運用方針（モノレポ vs 別リポジトリ）

- **別リポジトリに分けず、1つのリポジトリに複数ツールを入れる（モノレポ）方式を継続。**
  理由: 別リポジトリにするとツールの数だけリポジトリ・デプロイ設定・READMEの管理対象が増え、かえって混乱の元になる。1リポジトリなら「knacit-tools を開くだけ」で全体を把握でき、訪問者にも整理された印象を与えられる。
- **依存ライブラリの混在を防ぐため、`package.json` はツール直下（`tools/<名前>/`）に置き、ルートには置かない。**
  理由: ルート直下に1つ置くと全ツールの依存関係が1つの箱に混ざり、片方の更新がもう片方を壊す事故が起きる。
- **`main` ブランチには完成したツールだけを置く。未完成のものは公開しない。**
  理由: ツールを「使うだけ」の人はデプロイ先URLしか見ないので影響しないが、リポジトリ自体を見に来る人（採用担当・エンジニア等）には全ツールが見える。作りかけが混ざると、良いツールの評価まで下がるため。
- **将来、特定のツールが独り立ち（専用ドメイン・npm配布・単体での継続開発）したら、別リポジトリへの引っ越しを検討する。**
  理由: 最初から分ける理由はないが、育ったツールは分離した方が扱いやすくなる。

## 2026-07-16 — リポジトリの土台

- **AGENTS.md を本体、CLAUDE.md をシンボリックリンクにする。**
  理由: AGENTS.md は Linux Foundation（Agentic AI Foundation、Anthropic/OpenAI/Block 共同設立）管理の業界標準で、6万以上の公開リポジトリが採用。どの AI ツール（Claude Code / Cursor / Copilot 等）からも同じルールが読める。
- **AI 向け設定ファイルは公開する。**
  理由: 海外の主要 AI 企業（OpenAI 自身のリポジトリに88個の AGENTS.md）が公開しており、整理された設定はむしろ技術力のシグナルになる。ただし秘密情報は絶対に含めない。
- **`.claude/settings.json` はコミット、`settings.local.json` は gitignore。**
  理由: 共有設定と個人設定を分けるのが Claude Code の公式推奨パターン。local のコミットは最頻出ミスとされる。
- **ツールごとにネストした AGENTS.md を置く。**
  理由: 作業ディレクトリに応じて自動で文脈が切り替わり、`/clear` 後も引き継がれる。OpenAI のモノレポと同じ方式。
- **リポジトリ名は knacit-tools。**
  理由: 複数の便利ツールを継続的に置く前提。lab（実験場）より tools の方が用途が伝わる。
